git记录

Posted on 2017-02-22 | In 编程 | Views:

总结了基本git全部的用法，再也不怕以后忘了……

准备

第一次使用git时，要做以下的事情：

申请密钥（windows在用户名下的.ssh文件夹下），对当前电脑有效:

ssh-keygen -t rsa -C "youremail@example.com"

如果一切顺利的话，可以在用户主目录里找到.ssh目录，里面有id_rsa和id_rsa.pub两个文件，这两个就是SSH Key的秘钥对，id_rsa是私钥，不能泄露出去id_rsa.pub是公钥，可以放心地告诉任何人。至于为何这样，因为GitHub需要识别出你推送的提交确实是你推送的，而不是别人冒充的，而Git支持SSH协议，所以，GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。

配置参数如下:

注意git config命令的–global参数，用了这个参数，表示你这台机器上所有的Git仓库都会使用这个配置，当然也可以对某个仓库指定不同的用户名和Email地址。

git config --global user.name "Your Name"
git config --global user.email "email@example.com"

之后就可以开始操作git了

各种操作集合

初始化

git init命令用于初始化代码仓库。当然也可以不自己创建，去偷学习别人的。

创建一个本地仓库的克隆版本：
git clone /path/to/repository

如果是远端服务器上的仓库，你的命令会是这个样子：
git clone username@host:/path/to/repository

git基本结构

本地仓库有三棵树，第一棵是文件夹，也就是windows下的目录，也是工作目录（工作区有一个隐藏目录.git，这个不算工作区，而是Git的版本库，也就是后两者）；第二棵是缓冲区（stage或index），临时保存你的改动；最后一个是HEAD，指向最后一次commit，master分支。

第一步当然是用git add <filename>或git add * 命令来添加到缓冲区。
第二步是git commit -m "，这时你的改动已经提交到了 HEAD，一次性把暂存区的所有修改提交，-m后面输入的是本次提交的说明（字符串），可以输入任意内容，当然最好是有意义的，这样你就能从历史记录里方便地找到改动记录（用git log命令，记录commit到HEAD内的，不是缓冲区中的。如果嫌输出信息太多，看得眼花缭乱的，可以试试加上--pretty=oneline参数）。

时光旅行

运用git status命令可以随时查看当前仓库的状态，比如有啥文件修改了没提交等等。如果想很明确知道文件哪里修改了，那就用git diff 文件名吧，它会告诉你呦。

这时，我想回退了。我就可以用git reset命令。在git中用HEAD表示当前版本，上一个版本就是HEAD^,上上个版本就是HEAD^^……。往上100个版本就可以写成HEAD~100。想到那一个版本就用git reset --hard HEAD或commit一栏的id号，HEAD就是个指针。git提供了git reflog命令来让你查看历史记录（关于版本跳转的），第一列就是commit id。

需要注意，git保存的是修改而不是文件，如果修改了源文件而不add直接commit，修改不会被提交，用git diff HEAD -- 文件名就可以看区别。所以，修改了源文件一定经过add再commit！

git checkout -- 文件名可以让你的源文件回退修改，如果自修改以来没有提交到暂存区，那么撤销就是回到版本库的状态；如果已经提交到暂存区才修改的，那么只会回退到暂存区的版本。

git reset HEAD 文件名可以让你把暂存区的修改清空，HEAD表示当前版本。然后可以用上面的命令再恢复源文件。

git rm 文件名用于删除一个文件。和在工作目录删除文件是两码事。

远程仓库————github登场

在github建立仓库后，用git remote add origin ssh或http即可关联远程仓库，这里远程库的名字就是origin。

git push <远程主机名> <本地分支名>:<远程分支名>命令用于推送到远程仓库。如果省略远程分支名，则表示将本地分支推送与之存在”追踪关系”的远程分支（通常两者同名），如果该远程分支不存在，则会被新建。

如果省略本地分支名，则表示删除指定的远程分支，因为这等同于推送一个空的本地分支到远程分支。

git push origin :master
# 等同于
git push origin --delete master删除origin主机的master分支

如果当前分支与远程分支之间存在追踪关系，则本地分支和远程分支都可以省略。

git push origin

如果当前分支与多个主机存在追踪关系，则可以使用-u选项指定一个默认主机，这样后面就可以不加任何参数使用git push。

git push -u origin master

上面命令将本地的master分支推送到origin主机，同时指定origin为默认主机，后面就可以不加任何参数使用git push了。由于远程库是空的，我们第一次推送master分支时，加上了-u参数，Git不但会把本地的master分支内容推送的远程新的master分支，还会把本地的master分支和远程的master分支关联起来，在以后的推送或者拉取时就可以简化命令。

不过我第一次却失败了，用下面的force才行或者用git pull --rebase <远程主机名> <远程分支名>:<本地分支名>命令同步后再push，这时工作目录和远程一样一样的。

如果远程主机的版本比本地版本更新，推送时Git会报错，要求先在本地做git pull合并差异，然后再推送到远程主机。这时，如果你一定要推送，可以使用–force选项。

git push --force origin

分支管理的艺术

每次提交，Git都把它们串成一条时间线，这条时间线就是一个分支。HEAD严格来说并不是指向commit提交而是指向当前分支。
git branch 分支名创建一个分支，git branch会列出所有分支，加*号的是当前分支。git branch -b 分支名相当于：

即创建并切换分支。

git merge 分支名用于合并分支到当前分支,之后可以用git branch -d 分支名删除被合并的分支。

Git鼓励你使用分支完成某个任务，合并后再删掉分支，这和直接在master分支上工作效果是一样的，但过程更安全。

通常，合并分支时，如果可能，Git会用Fast forward模式（直接更新到最新的分支），但这种模式下，删除分支后，会丢掉分支信息。如果加上--no-ff参数，会在合并时进行commit，这样会从历史中git log --graph可以查看历史合并分支，由于进行了一次commit，最后加上-m参数。
有时候合并会冲突，如果要合并的两个分支中同一个文件同一个地方不同的话，需要手动修改好在add、commit即可。
